This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

A DIY bus departure display: a Waveshare 7.5" e-paper panel driven by an ESP32, running MicroPython, showing real-time departures from Stockholm's SL Transport API. The spec is discovered incrementally with the owner — this file is the source of truth for decisions already made. **When a design decision is settled in conversation, record it here** (update "Decisions" / remove it from "Open questions").

You cannot see the physical screen. Verification loop: deploy → reset → watch serial output → ask the owner to look at the panel. Make the code corroborate the screen: whenever the display is redrawn, print the same content as text to serial so logs alone can confirm most behavior.

## Hardware (fixed facts — do not rediscover these)

- **Board**: Waveshare "Universal e-Paper Driver Board" — ESP32-WROOM-32, 4MB flash, no PSRAM. Appears as a COM port over USB.
- **Panel**: Waveshare 7.5" e-Paper **V2**, 800×480, black/white only, SPI, controller embedded on panel. Full refresh takes ~4–5 s and flashes black/white; that is normal.
- **Panel-to-ESP32 wiring is fixed by the board** (non-standard SPI pins — must be set explicitly, defaults won't work):
  - BUSY=25, RST=26, DC=27, CS=15, SCK/CLK=13, MOSI/DIN=14 (no MISO — display is write-only)
  - `SPI(2, baudrate=4_000_000, sck=Pin(13), mosi=Pin(14))` works; ESP32 GPIO matrix allows any pins.
- **7.5" V2 BUSY is active-low, and polling the pin alone is not enough**: the panel is busy while BUSY reads 0, and the reference driver sends command `0x71` (status read) before *each* read of the pin inside the wait loop — without the 0x71 the flag doesn't update and the wait can hang forever.
- **Pixel polarity (7.5" V2, confirmed byte-for-byte against the reference source, not just its docstring)**: at the wire, bit **0 = white, 1 = black** — `Clear()` sends `0x00` bytes on the `0x13` plane to whiten the screen; the driver's `getbuffer()` XOR-inverts PIL data (where 1=white) before sending. So with `framebuf.FrameBuffer(buf, 800, 480, framebuf.MONO_HLSB)`: `fill(0)` = white background, draw with color 1 = black. Still confirm with a test pattern on first bring-up — see `tools/test_pattern.py`.
- **Every full refresh writes TWO data planes, not one**: command `0x10` gets the bitwise-NOT of the image (the "old"/previous-image plane the controller needs internally), then command `0x13` gets the real image, then `0x12` triggers the actual refresh. Sending only `0x13` is not sufficient — `Clear()` and `display()` in the reference driver both always write both planes before `0x12`.
- **Reference driver already ported** — see `src/epd7in5v2.py`. It's a line-for-line MicroPython translation of Waveshare's `epd7in5_V2.py` (fetched raw from https://github.com/waveshareteam/e-Paper, RaspberryPi_JetsonNano/python/lib/waveshare_epd/, 2026-07-04): same command bytes/order for `init()`, `clear()`, `display()`, `sleep()`, same two-plane write, same `0x71`-before-each-busy-read polling. **Do not "improve" or re-derive the init sequence** — if the screen misbehaves, the bug is almost certainly in wiring, framebuffer content, or call order, not in the command bytes. `init_fast`/`init_part`/4-gray/partial-refresh from the reference driver are intentionally not ported (see Open questions) — add them only when actually needed. **Already confirmed working on real hardware** (2026-07-04): flashed, wired, ran `tools/test_pattern.py` — owner confirmed sharp, correctly-positioned, correctly-polarized output.
- **No ready-made MicroPython driver exists for this panel+board combo — checked, don't re-search for one.** Waveshare's own repo (link above) only has Arduino/STM32/RaspberryPi_JetsonNano folders, no MicroPython. The most common community port, https://github.com/mcauser/micropython-waveshare-epaper, has `epaper7in5.py` but no V2 variant (predates this panel revision's different init sequence). The port in `src/epd7in5v2.py` — translating Waveshare's own RPi driver — **is** the "use Waveshare's solution" path for this hardware; it's not a workaround. The pin mapping came from the GxEPD2 Arduino reference for this exact board (also cited above), not guesswork, and is now hardware-confirmed too.
- RAM budget: **165,632 bytes free heap measured** right after boot (`gc.collect(); gc.mem_free()`) on this board with MicroPython v1.28.0 ESP32_GENERIC, before any app code runs.
- **The 48 KB framebuffer must NOT be held permanently resident — allocate it only for the brief draw+refresh window, then let it be freed.** This reverses earlier guidance in this file (which said to allocate it once, first thing, and keep it for the process lifetime) — that was right for RAM safety alone, but wrong once HTTPS entered the picture. Measured directly (`main.py`, 2026-07-09): with the framebuffer resident, the HTTPS/TLS handshake to SL's API fails or hangs *every time*, even with 75-90 KB nominally free (mbedtls's RSA-2048 certificate handling for this host needs more contiguous room than that; repeated `gc.collect()` doesn't recover it — it's live object memory, not fragmentable garbage). Freeing the framebuffer (drop all references, `gc.collect()`) before each network fetch and only allocating a fresh one when actually drawing+pushing to the panel fixed this reliably — see `main.py`'s `_draw_and_refresh()`. This is safe (not a repeat of the two-buffers gotcha below) as long as exactly one 48 KB buffer is ever alive at a time: allocate, use, drop, collect, repeat.
- **Never hold two BUF_SIZE (48,000-byte) buffers alive at once — MicroPython's allocator doesn't defragment.** Hit this directly: `epd7in5v2.py`'s first `display()` implementation allocated a second 48 KB buffer (for the inverted "old" plane) while the framebuffer was still alive, and it threw `MemoryError` despite ~165 KB nominally free — a large contiguous block can fail to find room even when the sum of free memory looks fine, because freed/live allocations of varying sizes fragment the heap and nothing compacts it. Fix (already applied): stream large writes through a small reusable scratch buffer (`_write_bulk_inverted`/`_write_fill` in `epd7in5v2.py`, chunked at 512 bytes) instead of ever allocating a second full-size buffer. Apply the same pattern to any future code that transforms the framebuffer (e.g. partial-refresh diffing) — chunk it, don't duplicate it.

## Physical mounting & drawable area (settled 2026-07-09)

The panel is mounted rotated 90° for portrait viewing, inside a picture frame whose window crops the panel's edges unevenly. All content must go through a 90-degree coordinate transform from a "logical" portrait canvas into the native 800×480 physical framebuffer, and stay within a calibrated safe sub-rectangle — not the full logical canvas.

- Logical portrait canvas: 480 wide (`LW`) × 800 tall (`LH`) — swapped from the native 800×480 physical buffer.
- Transform: for a logical point `(lx, ly)`, physical buffer coords are `(px, py) = (WIDTH - 1 - ly, lx)` (`WIDTH` = 800). A circle keeps its radius; an ellipse's radii swap; a rect's origin/size transforms as `physical.fill_rect(WIDTH - ly0 - lh, lx0, lh, lw)` for logical rect `(lx0, ly0, lw, lh)`. Framebuf can't draw rotated text directly — render into a small scratch buffer sized to the string, then copy its set pixels through the same transform (see `display.py`'s `_text()`, the production implementation of this pattern). Confirmed correct on the physical panel (an early smiley-face + text smoke test, then the calibration rectangle below, then continuously by the real app every refresh since).
- **Safe drawable rectangle**, calibrated against the actual frame with `tools/calibration_guide.py` and confirmed against the physical panel: within the 480×800 logical canvas, margins from each logical edge are **left=7px, top=33px, right=0px, bottom=43px**. Lay content out inside this inset rectangle, not the full canvas — right is 0 because that side had almost no crop to begin with.
- **Never draw a visible border/outline around the drawable area in the real app** — the owner found a border makes any minor misalignment obvious and distracting. Use the margins above as silent layout bounds, not as a rendered frame.

## E-paper rules (breaking these damages the panel or wastes hours)

1. Call the driver's `sleep()` after every refresh. Leaving the panel powered in an active state degrades it.
2. Full refresh only, and only when displayed content actually changed (compare rendered text, not raw API responses — `expected` timestamps jitter). Departure minutes change ~1×/min; that's the natural cadence. Waveshare conservatively recommends ≥180 s between refreshes — the owner tunes this trade-off, don't silently decide.
3. Init → write buffer → display/turn-on → sleep, every cycle. After `sleep()` the panel needs re-init before the next refresh.
4. The refresh is a blocking busy-wait (~5 s). Acceptable; don't try to make the SPI driver async.

## Toolchain & commands

Host tools (install once): `pip install esptool mpremote pytest mpy-cross` (mpy-cross compiles vendored libraries to bytecode — see MicroPython/ESP32 gotchas below for why this matters)

```
mpremote connect list                          # find the COM port (call it COM3 below)
```

One-time firmware flash (download ESP32_GENERIC .bin from micropython.org/download/ESP32_GENERIC — plain 4MB variant, not SPIRAM/OTA):

```
esptool --port COM3 erase_flash
esptool --port COM3 --baud 460800 write_flash 0x1000 ESP32_GENERIC-<version>.bin
```

(esptool v5 prefers hyphenated `erase-flash`/`write-flash`; underscore aliases still work. If it can't connect, hold the board's BOOT button while it retries.)

Everyday loop:

```
cd src && mpremote connect COM3 fs cp -r . :   # deploy everything (bash; older mpremote may reject "cp -r ." — then copy files/dirs explicitly)
mpremote connect COM3 fs cp src/main.py :main.py   # deploy a single changed file (faster, prefer this)
mpremote connect COM3 reset                    # restart so new code runs (cp does NOT restart)
mpremote connect COM3 repl                     # serial console; Ctrl-] exits, Ctrl-D soft-reboots, Ctrl-C interrupts main.py
mpremote connect COM3 run tools/somescript.py  # run a host-side file on the device WITHOUT copying — ideal for hardware experiments
mpremote connect COM3 fs ls                    # verify what's on the device
pytest                                         # host-side tests for the pure-logic modules
```

Gotchas: only one process can hold the COM port — close any open REPL before deploying. `main.py` auto-runs on boot; structure it so Ctrl-C reaches the REPL (catch exceptions, print, idle — never crash into a tight reset loop, or the board becomes hard to reflash).

## Architecture

Single asyncio event loop (`asyncio` in MicroPython). No threads. **The admin web server and the departure-fetch/redraw cycle are NOT both run once connected to home Wi-Fi** — see "Departures logic & stops" for why; this reverses the original "two long-lived tasks always on" design.

Boot flow: load `/config.json` → try Wi-Fi STA with stored creds (~15 s timeout) → on success: NTP sync, brief settle delay, then go straight into the departure-fetch/redraw loop (no separate "booted" screen, no admin server). On failure: start AP mode (SSID `BusDisplay-Setup`, portal at `http://192.168.4.1`) serving a form for SSID/password → save to `/config.json` → reboot. Wi-Fi creds therefore only ever exist on the device, never in the repo. The framebuffer is NOT allocated at boot — see "Hardware" RAM budget notes; it's allocated transiently, only when actually drawing.

Intended layout (`src/` maps 1:1 to the device filesystem root; adjust as the project evolves):

```
src/
  main.py          # entry: boot flow, transient framebuffer alloc, asyncio loop
  config.py        # load/save /config.json on device
  wifi.py          # STA connect with timeout, AP fallback
  server.py        # admin panel + config API (Microdot)
  display.py       # layout/rendering onto the framebuf; also logs what it drew
                   #   (keep layout sectioned — departures are the first content source, not the only one; see Future direction)
  epd7in5v2.py     # low-level panel driver (port of Waveshare's, pins above)
  sl.py            # thin I/O wrapper: fetch departures JSON over HTTPS
  departures.py    # PURE logic: parse/filter/format SL JSON — no hardware imports
  lib/             # vendored third-party (microdot.py, writer.py, generated fonts) — don't hand-edit
tests/             # pytest, runs on host CPython
tools/             # host-side scripts (font generation, one-off device experiments)
```

**Testability rule**: anything that can be pure (parsing, filtering, formatting, layout math) goes in modules with no `machine`/`network`/`requests` imports at top level, so it runs under host CPython with pytest. Hardware and network stay in thin adapters. This is the only automated testing this project has — on-device behavior is verified by eye.

Key library choices (decided): **Microdot** (single-file asyncio web framework, vendor into `src/lib/`) for the admin panel — **only runs during AP-mode Wi-Fi setup now, not once connected** (see "Departures logic & stops" for why); **peterhinch/micropython-font-to-py** `writer.py` + generated font modules for large text — framebuf's built-in 8 px font is unreadable at display distance and is ASCII-only (see "Departures logic & stops" for the Swedish-character workaround in the meantime); generate ~40–64 px fonts for departure lines. Admin panel HTML: one small page, inline CSS, no CDN/build step (device may be offline). Current display code uses a scaled-up built-in font (`display.py`, `TEXT_SCALE`/`HEADER_SCALE`) as a stopgap — legibility at actual viewing distance not yet confirmed by the owner.

## SL Transport API (verified 2026-07)

- Departures: `GET https://transport.integration.sl.se/v1/sites/{siteId}/departures` — **no API key**. Query params: `transport=BUS`, `forecast=<minutes>`, `line`, `direction` (`transport`+`forecast` verified live 2026-07). Keep the JSON small on-device — a busy stop returned 45 departures at `forecast=60`; use `line`/`direction` filters and/or a smaller `forecast`.
- Each departure has `destination`, `line.designation`, `direction_code`, `stop_point`, `scheduled`/`expected` (ISO, local time, no tz offset) and **`display`** — a preformatted string ("Nu", "5 min", "12:34"). **Show `display`**; it sidesteps CET/CEST timezone math entirely (NTP gives UTC only).
- Finding the siteId: `GET /v1/sites` is ~MB-sized — fetch it on the **host** (curl/browser), never on the device. Note SL siteIds differ from the old Stop Lookup API ids. Site ids live in code (`main.py` config block), not an admin panel — see "Departures logic & stops".
- Test from host: `curl "https://transport.integration.sl.se/v1/sites/9192/departures?transport=BUS&forecast=60"` (9192 = Slussen).
- Poll every 30–60 s. No key means shared fair-use quota — don't hammer it.
- Docs: https://www.trafiklab.se/api/our-apis/sl/transport/

## Departures logic & stops (settled 2026-07-09)

- **Config lives in `/settings.json`, deliberately gitignored** (`src/settings.json` in `.gitignore`): stop ids, direction filter, forecast window, refresh cadence. Initially these were plain constants in `main.py`, but the owner didn't want their home stop identifiable in a public repo (siteId narrows down roughly where they live) — so it moved to a JSON file loaded at runtime (`settings.py`, mirrors `config.py`'s pattern), not committed. `src/settings.example.json` **is** committed as the template/field reference (uses the public Slussen siteId as a placeholder, not the owner's real stop). Workflow: copy the example to `src/settings.json`, fill in real values, deploy like any other file (`mpremote fs cp src/settings.json :settings.json`) — no code change or redeploy of `main.py` needed for settings-only changes. `config.py` stays Wi-Fi-only (also gitignored in spirit — device-only, never even written to the host repo). JSON not YAML: MicroPython has no YAML parser and `json` is already built in and already used for `config.json`.
- **No "primary"/"fallback" concept anymore (removed 2026-07-09, owner's call)**: `settings.json`'s `"stops"` is a flat, ordered list of `{"name": ..., "site_id": ...}` objects — "just stops I care about, thats it." No suitability logic, no picking one over another. **Every** configured stop is always fetched and always shown, in the list's order. `departures.is_suitable()`/`minutes_until()` were deleted outright as dead code, not deprecated in place. (Stop names briefly weren't in config at all — "we dont even need the names" — but that changed same-day once the owner asked for the name to be shown on screen as a section label; see "Screen design" below. Names are fine in `settings.json` since that file is gitignored; only `settings.example.json`, which is committed, uses a placeholder.)
- **Departures per stop**: shows the next `departures_per_stop` (default 3) departures for **each** stop, regardless of how far out ("say 12 or 24h window"). In practice: SL's `forecast` query param caps at **1200 minutes (20h) — confirmed by hitting a 400 Bad Request past that**, and a busy hub like Slussen returns ~31KB at that window, which is a real RAM risk (see "MicroPython/ESP32 gotchas" — a JSON parse needs ~2-3x its size in free RAM). Defaulted `forecast_min` to **180 (3h)** instead — comfortably enough to find 3 departures at any real stop without the payload-size risk of the full 20h window the owner floated as an example; raise it in `settings.json` if a very sparse-service stop ever comes up empty.
- **Direction filter**: only show buses towards Gustavsberg/the city — SL's `direction_code=2` (confirmed: both stops' `direction_code=2` departures go to "Gustavsbergs centrum"/"Slussen"; `direction_code=1` goes the other way, e.g. "Eknäs"/"Idalen"). Filtered server-side via the `direction` query param (`sl.fetch_departures(..., direction=cfg["direction_code"])`) — keeps the response small, per the API's own documented filters (see "SL Transport API").
- **Fetch strategy**: every stop in the list is fetched independently and unconditionally each poll cycle (`main.py:_fetch_all_stops`) — one stop's fetch failing doesn't blank the others; that stop just keeps showing its last-known-good departures with the overall footer marked stale (see below). "Updated" only advances when **all** stops succeed in the same cycle.
- **Ordering bug, fixed**: departures used to sort by `scheduled`, which could show a delayed bus's clock-time departure *before* an on-time bus's sooner relative-time one — a rider would actually see the on-time bus first. Fixed by sorting by `expected` instead (SL's live prediction, which `display` is computed from) — see `departures.parse_departures()` and the regression test `test_parse_departures_sorts_by_expected_not_scheduled_when_delayed` in `tests/test_departures.py`.
- **Refresh cadence — owner's explicit call, not the conservative default**: poll SL every `poll_interval_s` = 60 s, and push an actual panel refresh whenever the rendered text changed and `min_refresh_interval_s` (also 60 s) has passed. The footer shows live relative countdowns ("N min") which go stale within a minute, so the owner asked for ~1×/min panel refreshes — this is *well below* Waveshare's conservative ≥180 s guidance (see "E-paper rules"), a deliberate, informed tradeoff, not a silent regression. **Also see "Screen refresh strategy" below** — a full flicker every single minute is exactly the kind of thing the owner is now weighing against panel longevity; nothing implemented yet, discussion only.
- **Swedish characters (å/ä/ö/é/ü) corrupt on-screen** — confirmed via serial log ("Eknäs" → garbled) — because framebuf's built-in font is ASCII-only. `departures.parse_departures()` transliterates to plain ASCII (`_to_ascii` in `departures.py`) as a stopgap. Generating a proper font via peterhinch/micropython-font-to-py (already a declared "Key library choice") will fix this properly and isn't done yet.
### Screen design (settled 2026-07-09)

The guiding principle, stated by the owner and worth keeping in mind for any future layout change: **this is a kitchen-counter, glance-from-across-the-room display, not a reading surface.** The single fact that matters most in that use case is "how soon do I need to leave" — not the destination, not the line number. Everything else in this section follows from designing around that one fact, not from generic "make text bigger" tweaks.

- **Per-stop sections, clearly separated**: each configured stop gets its own section, labeled with its name (`display.stop_section()`), in `settings.json` order. Between sections: a thicker rule (`RULE_HEIGHT` = 3) plus a deliberately generous gap (`GROUP_GAP` = 34) — "separate the stops more, so its visually distinct" (owner). This is an internal content divider, not the "border around the drawable area" that's banned (see "Physical mounting & drawable area") — it sits between sections, never around the whole screen.
- **Hero + caption, not a bigger row**: each stop's next (soonest, `expected`-sorted) departure is not just "a bigger version of the row format" — its countdown/time string (`display`, e.g. "3 min", "Nu", "18:12") is drawn alone, huge (`HERO_SCALE` = 7, centered), because that string alone answers the one question that matters at a glance. A small caption underneath (`CAPTION_SCALE` = 2, centered, "`<line>  <destination>`", no fixed-width padding since padding would break the centering math) answers "which bus" for anyone who looks a beat longer. The stop's other departures are a compact left-aligned list at `ROW_SCALE` = 2, using the existing full 3-column format (line/destination/display) — secondary information, not competing for attention with the hero.
- **Footer, bigger and lower (revised 2026-07-09)**: an earlier pass used `FOOTER_SCALE` = 1 (the font's native 8px) and the owner found it "hard to read" — bumped to `FOOTER_SCALE` = 2. At that size "Updated HH:MM   Weekday D Mon HH:MM" no longer fits one row at `CONTENT_W`, so it now wraps to two (still handled by `display.footer_lines()`'s existing fits-one-row-else-two logic) — a deliberate, accepted tradeoff for legibility over the "footnote, one row" ideal from the previous pass. It's also centered and anchored with its own smaller `FOOTER_MARGIN` (12px, vs. the main content's 25px `CONTENT_MARGIN`) so it sits closer to the true hardware-safe bottom edge than before ("move it lower" — owner) without needing to touch the calibrated crop-safety margins, which are a hardware fact, not a styling knob. "Updated" only advances when every configured stop's fetch succeeds that cycle, so it visibly falls behind "now" when anything is stale — that gap, plus an explicit "(stale)" suffix, is the staleness indicator. Local (Stockholm) time is computed from the NTP-synced UTC clock via `localtime.py`, a pure, dependency-free CET/CEST (EU DST rule) converter with its own host tests (`tests/test_localtime.py`) — MicroPython has neither `datetime` nor a tz database, so this hand-rolls day-of-week (Sakamoto's algorithm) and calendar rollover arithmetic; deliberately avoids `time.mktime`/`time.localtime` for portability between host pytest and the device.
- **Content margin**: an additional ~5 mm (`CONTENT_MARGIN` = 25px, approximate at the panel's ~0.204mm/px published dot pitch — trust a physical ruler over this number) inset within the calibrated safe rectangle from "Physical mounting & drawable area" — that rectangle is the hardware crop-safety fact; this margin is a separate styling choice (owner: "its very much on the edge right now") applied on all four sides via `display.py`'s `CONTENT_X0/Y0/W/H`, except the footer's bottom edge (see above).
- **Not done, considered and set aside for now**: merging all stops' departures into one globally-time-sorted list (rejected — owner explicitly wants clear per-stop sections, not a blended feed); mixing font scales within a single line (e.g. huge digits + smaller "min" suffix in one string) for the hero, which would look more like a real countdown clock — technically possible with the existing per-call `scale` parameter in `_text()`, but adds real positioning complexity for a marginal gain over "draw the whole hero string big"; not built, worth revisiting if the current hero treatment doesn't read as glanceable enough once seen physically.

### Screen refresh strategy — open question, discussion only, nothing implemented (2026-07-09)

The owner asked whether a full-flicker refresh is really needed every ~1 min, or whether something smarter exists for both look and panel longevity. Worth recording the state of that thinking here since it'll come up again:

- The Waveshare reference driver this project already translated (`src/epd7in5v2.py`) has **`init_part`/partial-refresh support that was deliberately not ported** (see "Hardware" — "add them only when actually needed"). Partial refresh updates a sub-rectangle without the full black/white flash, so it's faster and much less visually disruptive.
- The tradeoff: partial refresh typically can't do the full clear-and-rewrite the panel needs to fully discharge each pixel, so **ghosting accumulates** with repeated partial updates — most e-paper vendors (Waveshare included, based on how `init_part`/`init_fast` are typically documented for this controller family) recommend an occasional full refresh to clear it, even in a partial-refresh-heavy design.
- Given this screen's actual content shape — a small footer clock ticking every minute, versus departure rows that only change every few minutes — a **hybrid** approach seems like the natural fit: partial-refresh just the footer's rectangle every minute (cheap, low-flicker, satisfies "always show a fresh clock"), and reserve full refreshes for when the departure content itself actually changes, or on some periodic cadence (e.g. every N minutes or every N partial refreshes) specifically to clear accumulated ghosting.
- This is **not yet implemented** — it would mean porting `init_part` and a partial-window `display()` variant from the Waveshare reference (same translate-don't-re-derive discipline as the rest of `epd7in5v2.py`), plus tracking a ghosting-clear cadence in `main.py`. Worth doing if the owner decides ~1×/min full refreshes are visibly bothersome or a real longevity concern in practice; not done speculatively.

### The RAM-vs-HTTPS conflict — full story, corrected understanding (read before touching main.py's boot flow)

Extensive on-device testing (2026-07-09) found the departure fetch can fail to coexist with other resident memory on this board. **The investigation happened in two rounds and the first round's conclusion was incomplete — read both.**

**Round 1 finding (still true, but not the dominant cause)**: with the 48 KB framebuffer allocated once at boot and kept resident (as this file used to recommend), the HTTPS/TLS handshake to `transport.integration.sl.se` failed or hung very reliably — `MBEDTLS_ERR_PK_INVALID_PUBKEY`, `MBEDTLS_ERR_X509_ALLOC_FAILED`, `MBEDTLS_ERR_RSA_PUBLIC_FAILED+MBEDTLS_ERR_MPI_ALLOC_FAILED` (an allocation failure during the RSA-2048 certificate's public-key math — SL's cert is Amazon-issued RSA 2048, confirmed via `openssl s_client`), or a hard `abort()`/reboot — even with 75-92 KB nominally free; repeated `gc.collect()` didn't recover it (it's live object memory, not garbage) and `ssl.CERT_NONE` didn't help (mbedtls still parses/uses the RSA key regardless). **Fix, still in place**: don't keep the framebuffer resident — allocate it only for the brief draw+`epd.display()` window (see "Hardware" RAM notes, `main.py`'s `_draw_and_refresh()`).

**Round 2 finding — the actual dominant cause**: after the framebuffer fix, first-boot fetches were *still* failing/hanging on nearly every fresh boot (6/6 in one stretch), even with the admin server's `start_server()` never called in STA mode. Root cause: `main.py` still did `import server` unconditionally at the top of the file, and `server.py` builds its whole Microdot app + route table (`app = Microdot()`, `@app.route(...)`) **at import time** — so merely importing the module cost the same ~30 KB+ that round 1 already showed was fatal to the TLS handshake, regardless of whether the server was ever started. **Fix**: `import server` moved to be lazy, inside the AP-mode branch of `main()`, where it's actually needed — never imported at all in STA/connected mode. Confirmed this was the real fix: an isolated script reproducing main.py's exact sequence (connect → NTP → 3 s settle → fetch) succeeded reliably both outside and inside `asyncio.run()`; only the real `main.py`, with its `import server`, failed — and moving that import fixed it. **Lesson for future modules**: importing something is not free just because you don't call it, if its top-level code does real work (route registration, object construction) — check for that before assuming an unused import is harmless on this board.
- **Two socket-level mitigations remain from round 1, still worth keeping but demonstrably not sufficient on their own**: a 3 s settle delay after Wi-Fi connect, and an explicit 15 s `timeout` on every `requests.get()` call (`sl.fetch_departures(..., timeout_s=15)`). Confirmed via repeated testing that this timeout does **not** reliably bound every hang — one observed hang lasted 140+ seconds despite `timeout_s=15`, meaning it happened inside blocking work (likely the TLS handshake's own crypto processing) that a socket-level timeout doesn't cover. Don't rely on `timeout_s` alone as the safety net.
- **The actual safety net is a hardware watchdog**: `machine.WDT(timeout=WDT_TIMEOUT_MS)` (150 s) in `display_loop`, fed once per loop iteration before the fetch. If anything hangs long enough to miss a feed — regardless of cause, Python-catchable or not — the ESP-IDF task watchdog force-reboots the board. Confirmed working exactly as intended (observed a genuine 140+s hang get force-recovered via reboot, then the device resumed normally on the next boot). This is the right tool for "something might block in a way Python can't interrupt," and doesn't conflict with the "no threads" architecture (it's independent of asyncio/threading). A watchdog-triggered reboot is a controlled recovery, not the "tight reset loop" the toolchain notes warn against, as long as it only fires occasionally — if it starts firing on *every single boot*, that's a regression, revisit this section (and check for another eager unconditional import first).

## MicroPython/ESP32 gotchas

- `requests` (a.k.a. `urequests`): always `resp.close()` (in `finally:`), `gc.collect()` before each fetch, and never hold the raw response text longer than needed. HTTPS works; certs aren't validated by default (fine here).
- A JSON parse needs ~2–3× the response size in free RAM. If fetches start failing with `MemoryError`, shrink `forecast`/filter params first.
- Handle Wi-Fi drops: the fetch task should catch exceptions, keep the last good data on screen (with a stale indicator), and let a reconnect loop in `wifi.py` recover. Never let one failed request crash the program.
- **Vendor dependencies as precompiled `.mpy` bytecode, not raw `.py` — hit this directly, and the failure mode is misleading.** Deploying `src/lib/microdot.py` (58 KB source) as-is and then calling `network.WLAN(...).active(True)` right after raised `OSError: WiFi Out of Memory` with a `wifi nvs cfg alloc out of memory` C-level log line — nothing about the error pointed at the import. Root cause, confirmed by bisecting with `gc.mem_free()` before/after each step: compiling a large `.py` file *on the device* needs far more transient RAM than its final bytecode size (parser/AST overhead), and that transient spike fragmented the heap enough that the WiFi driver's own allocation failed afterward — even though `gc.mem_free()` looked fine again once the import finished (same non-defragmenting-allocator issue as the panel driver, see Hardware section). Fix: `pip install mpy-cross`, then `python -m mpy_cross -o src/lib/microdot.mpy src/lib/microdot.py`, deploy the `.mpy` instead of the `.py` (import statements resolve either extension transparently — no code changes needed). Compiled size dropped 58 KB → 12 KB and the same sequence then left ~77 KB free. Keeping both `microdot.py` (source, for regenerating the `.mpy`) and `microdot.mpy` in the repo/on-device is safe — confirmed on this firmware that MicroPython prefers the `.mpy` when both exist alongside each other, so the `.py` is just inert extra flash usage, not a re-trigger of the bug. **Apply this to any other large vendored library** (font modules, once generated, are likely candidates too). Note: PyPI's `mpy-cross` package (currently 1.27.0) lags the latest firmware release (1.28.0 here) — the bytecode format hasn't changed between them as of 2026-07, confirmed by testing on this exact board, but if a future `.mpy` fails to import with a version/format error, that's the first thing to check.

## Working conventions for this repo

- Prefer a 5-line experiment via `mpremote run` over speculation about hardware behavior. For driver/command-sequence questions, the primary source is Waveshare's own reference implementation (link above), fetched raw — not the wiki pages, which returned 403 when checked, and not a search engine's summary of them.
- Deploy small and often; debug with serial prints. Don't refactor the panel driver once it works.
- If the panel stays blank with correct-looking code, check in this order: (1) wiring against the pin mapping above, (2) that both `0x10` and `0x13` planes are being sent, (3) BUSY polarity/the `0x71` re-poll, (4) framebuffer color convention. No DIP switch or jumper for panel variant has been found on this board's own docs so far — don't go looking for one on the strength of unrelated Waveshare panel manuals; if one turns out to exist, record it here.
- **Don't trust a driver-code question answered by summarization** (e.g. a fetch tool that runs content through a small model). For exact command bytes/sequences, get the raw source and read it directly — a paraphrase is exactly the kind of "confidently wrong" detail that put earlier claims in this file at risk (see git history of this file).
